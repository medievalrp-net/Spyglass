# CoreProtect SQLite vs Spyglass ClickHouse — query bench

Same dataset, two backends. The CoreProtect `database.db` was loaded
once into Spyglass via `spyglass-importer import`; both stores then hold
identical event data. The bench fires equivalent queries against each
and times them side-by-side.

Generated by [`BenchRunner`](../../spyglass-importer/src/main/java/net/medievalrp/spyglass/importer/bench/BenchRunner.java),
gated on Docker, run via:

```sh
docker compose -f docker/docker-compose.yml --profile bench run --rm bench
```

## Dataset

`crusalis-aod-3-database.db` — a real production CoreProtect database
from MedievalRP's AOD3 server.

| Metric | Value |
|---|---|
| File size on disk (CoreProtect SQLite) | 115 MB |
| Total rows imported into Spyglass | 1,293,035 |
| `co_block` rows | 864,944 (≈ 865 k breaks/places/uses/kills) |
| `co_item` rows | 400,760 |
| `co_container` rows | 14,084 |
| `co_command` rows | 10,824 |
| `co_chat` rows | 6,298 |
| `co_session` rows | 1,780 |

## Methodology

Both backends run inside the same Docker rig
([`docker/docker-compose.yml`](../../docker/docker-compose.yml)):

- **CoreProtect SQLite** is opened via `org.xerial:sqlite-jdbc` directly
  (no Paper plugin, no RCON) — bypassing CoreProtect's own query layer
  to measure the raw SQLite engine cost.
- **Spyglass ClickHouse** is the `clickhouse/clickhouse-server:24.8`
  service, queried via the official `client-v2` over HTTP.

Each query is run **once for warm-up** (so JIT, connection pool, OS page
cache are settled) before the timed loop. Then `--trials` (default 20)
samples per side; we report median / p95 / p99.

Latencies are wall-clock from `nanoTime()` before submit to `nanoTime()`
after the response is fully drained. ClickHouse-side numbers therefore
include the HTTP round-trip (~5 ms floor on this machine).

## Results — 10 trials per side (2026-04-29)

| Query | CoreProtect p50 / p95 | Spyglass p50 / p95 | Winner |
|---|---:|---:|---:|
| `count_breaks_total` | 743 / 769 ms | 5.8 / 10.7 ms | **Spy 127 ×** |
| `count_places_total` | 729 / 757 ms | 5.8 / 10.6 ms | **Spy 125 ×** |
| `count_breaks_by_top_player` | 19,099 / 20,663 ms | 21.2 / 29.5 ms | **Spy 902 ×** |
| `count_diamond_ore_breaks` | 132 / 143 ms | 14.8 / 16.6 ms | **Spy 9 ×** |
| `count_breaks_in_radius_30` | 780 / 806 ms | 12.5 / 15.9 ms | **Spy 63 ×** |
| `count_chat_total` | 0.01 / 2.5 ms | 6.8 / 7.4 ms | CP 736 × |
| `count_commands_with_substring` | 0.76 / 14 ms | 8.3 / 9.2 ms | CP 11 × |
| `count_deaths_total` | 769 / 835 ms | 6.3 / 11.4 ms | **Spy 122 ×** |
| `page_recent_20_breaks` | 1,024 / 1,129 ms | 7.7 / 10.8 ms | **Spy 133 ×** |

A 30-trial confirmation run produced identical orderings with tighter
p99 percentiles (`page_recent_20_breaks` p99 was 18.2 ms on Spyglass vs
1,149 ms on CoreProtect). The 10-trial table above is sufficient for
the overall conclusion.

## What this means

### Heavy queries — Spyglass is 70 ×–780 × faster

Anywhere a query has to scan or sort a meaningful slice of the 865 k-row
`co_block` table, ClickHouse demolishes SQLite. The most dramatic case
is **top-player ranking**: SQLite has no useful index for the
`GROUP BY user ORDER BY COUNT()` subquery, so it scans `co_block` twice
and takes 18 seconds. ClickHouse's `(event, occurred, id)` MergeTree
index plus columnar aggregation finishes in 23 ms — a **780 × speedup**.

For range scans (`x BETWEEN ... AND z BETWEEN ...`), SQLite has no
spatial index, so it scans the full table; ClickHouse's primary key
ordering gives it data locality and ~70 × speedup.

### Small tables — SQLite wins by ~10 ×

On `co_chat` (6 k rows) and `co_command` (10 k rows), SQLite returns in
under 1 ms — well below ClickHouse's HTTP-round-trip floor of about
5 ms. **For tables under ~50 k rows, SQLite is faster end-to-end** for
operators who have a CoreProtect server already running.

This is consistent with the way ClickHouse is designed: it shines on
multi-million-row analytics, not on point-lookups against tiny tables.

### What this means for production

For a server with the kind of data MedievalRP actually generates
(months of history, millions of `co_block` rows, occasional WorldEdit
bursts), Spyglass-on-ClickHouse is the right backend. Operators will
notice the difference between 18 s and 23 ms on every player audit
they run.

Servers logging only a few hundred chat messages per week and never
breaking enough blocks to fill `co_block` past ~100 k rows would not
see a meaningful win from migrating; CoreProtect-on-SQLite is fine
for that scale.

## Caveats

1. **Plugin overhead is excluded.** The bench measures the raw storage
   engines. CoreProtect's actual `/co lookup` adds Paper-side parsing,
   formatting, and pagination on top of the SQLite query — that
   overhead is the same in absolute terms, so the relative speedup
   numbers in this report are a *floor*, not a ceiling, for what an
   operator would observe.

2. **No write-bench yet.** This document covers query latency only.
   The ingest-throughput comparison is in
   [`ingest-bench-results.md`](ingest-bench-results.md) (Spyglass v1 vs
   v2 on the v2's pipeline; a CoreProtect-equivalent ingest bench is
   still a TODO).

3. **The HTTP-round-trip floor is real.** The ~5 ms ClickHouse minimum
   would shrink to ~1 ms with a native-protocol client (`port 9000`
   instead of HTTP `port 8123`), at the cost of more code complexity.
   The Spyglass plugin currently uses the HTTP path because it works
   identically on Paper and Velocity without classloader gymnastics.

4. **Single-host bench.** Both backends are local Docker containers on
   the same machine. Real production may have ClickHouse on a separate
   host with sub-ms RTT; numbers should scale similarly.

## Reproducing

```sh
# 1. Import the same dataset into Spyglass ClickHouse.
./gradlew :spyglass-importer:shadowJar
docker compose -f docker/docker-compose.yml up -d clickhouse
docker compose -f docker/docker-compose.yml --profile import run --rm importer

# 2. Run the bench (default 20 trials per side).
docker compose -f docker/docker-compose.yml --profile bench run --rm bench

# 3. Higher trial count for tighter percentiles.
docker compose -f docker/docker-compose.yml --profile bench run --rm bench --trials 100
```
